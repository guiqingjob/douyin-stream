# Media Tools Web 端 UI/UX 重构与设计理念文档

## 1. 设计初衷与破局思路 (Design Philosophy)

在本次重构前，`media-tools` 的 Web 端（基于 Streamlit）面临着**“极客工具化”**向**“现代化 SaaS 产品”**过渡的阵痛：
- **状态耦合严重**：所有的页面导航依赖全局变量（单文件路由），导致浏览器原生功能（刷新、后退、链接分享）全部失效。
- **空间利用低效**：将“任务创建”、“进度查看”、“历史记录”和“文件管理”强行塞入同一个页面的不同 Tab 中，用户在操作时需要频繁横向切换，视觉焦点极易丢失。
- **阻塞式交互**：后台任务进度依赖全局页面重绘（`st.rerun()` + `time.sleep()`），导致严重的页面闪烁，打断了用户当前的其他输入操作。

### ✨ 核心设计理念：做减法、展空间、分主次

1. **Native Routing (回归原生路由)**：抛弃一切“自制框架”，全面拥抱 Streamlit 最新的 `st.navigation` 原生多页架构，让每个功能模块拥有独立的生命周期和 URL。
2. **Dashboard Layout (控制台式布局)**：将核心工作流页面从“垂直堆砌”改为“左右分栏”，实现**“左侧操作，右侧反馈”**的沉浸式体验。
3. **Asset Centralization (资产聚合)**：打破“下载管视频，转写管文稿”的孤岛效应，建立统一的“资产大盘”，实现文件状态的全局可视化。
4. **Asynchronous UI (无感刷新)**：引入局部渲染（`st.fragment`），让耗时的后台任务进度条在角落静默更新，绝不干扰用户的主线工作。

## 2. 页面结构与排版方案 (Layout Architecture)

### 2.1 全局导航结构 (Global Sidebar)
采用典型的分层分组侧边栏设计，建立清晰的心理模型：
- **核心工作流**：最常访问的生产力入口（仪表盘、下载中心、转写中心）。
- **资产管理**：统一的文件与结果查看入口（资产大盘）。
- **系统与配置**：低频的底层设置入口（关注管理、账号认证、存储清理、系统配置）。

### 2.2 核心页面布局：2:1 分栏设计 (Asymmetric Split)
在“下载中心”和“转写中心”中，我们彻底移除了原来的 Tab 结构，采用宽屏下的 `st.columns([2, 1])` 布局：

```text
+---------------------------------------------------------------------------------+
|                                                                                 |
|  [左侧主干区: 占宽 2/3]                               [右侧反馈区: 占宽 1/3]  |
|                                                                                 |
|  🚀 发起新任务 (输入表单)                             📌 当前任务状态           |
|                                                                                 |
|  🔗 输入视频链接:                                     +-----------------------+ |
|  [________________________________]                   | ⏳ 正在下载...        | |
|                                                       | 进度: [████░░░] 45%   | |
|  ⚙️ 选项设置                                          | 速率: 2.4 MB/s        | |
|  [x] 提取音频  [ ] 下载封面                           +-----------------------+ |
|                                                                                 |
|  [ 🚀 立即开始 (Primary) ]                                                      |
|                                                                                 |
+---------------------------------------------------------------------------------+
|  📜 底部通栏: 最近任务历史 (Task History)                                       |
+---------------------------------------------------------------------------------+
```
**设计优势**：用户在左侧点击“开始”后，视线自然右移即可看到进度。同时，左侧的表单立刻清空待命，用户可以**毫无阻碍地连续发起下一个任务**，而不需要切换回“当前任务” Tab。

### 2.3 资产大盘：聚合视图 (Asset Dashboard)
新建 asset_library.py（src/media_tools/web/pages/asset_library.py），作为所有本地产出物的最终归宿。
- **顶部数据卡片 (Metrics Cards)**：直观展示总数量、总占用、最近活跃时间。
- **底部数据表格 (Dataframe/Table)**：列表化展示素材和文稿。
- **跨模块联动 (Cross-linking)**：在视频素材库下方，直接提供 `[🎙️ 用这些素材去转写]` 的跳转按钮，打通了“下载 -> 转写”的业务断层。

## 3. 关键技术实现细节 (Technical Implementation)

### 3.1 基于 st.navigation 的原生路由
替换了原先庞大且脆弱的 `web_app.py` 中的 `if-elif` 逻辑，改用字典形式注册页面：
```python
pg = st.navigation({
    "核心工作流": [pg_home, pg_download, pg_transcribe],
    "资产管理": [pg_assets],
    "系统与配置": [pg_following, pg_accounts, pg_cleanup, pg_settings]
})
pg.run()
```
此举直接恢复了浏览器的前进/后退功能，并允许用户通过 `http://localhost:8501/download_center` 直接访问深层页面。

### 3.2 基于 st.fragment 的无闪烁进度轮询
这是解决页面崩溃与闪烁的核心技术。我们为轮询函数加上了 `@st.fragment(run_every=2)` 装饰器：
```python
@st.fragment(run_every=2)
def _poll_download_task() -> None:
    is_running = render_task_progress(empty_message="当前没有正在执行的任务")
    if not is_running:
        st.rerun() # 任务结束后，再进行一次全局刷新以更新历史记录
```
**技术优势**：
- Streamlit 每隔 2 秒**仅重新执行该片段的代码**。
- 左侧用户正在输入的文本框不会失去焦点。
- 彻底抛弃了阻塞主线程的 `time.sleep()`。

### 3.3 异步事件循环安全 (Asyncio Thread Safety)
在转写中心（src/media_tools/web/pages/transcribe_center.py）的后台线程中，原先直接在 `for` 循环中调用 `asyncio.run()`，这在密集并发下极易导致 `RuntimeError: Event loop is closed`。
重构后，将整个批量任务包装在一个外层的 `async def _run_batch():` 协程中，由一个全局的 `asyncio.run()` 启动，确保了底层的异步 HTTP 客户端（如 Qwen API）能安全复用连接池。

## 4. 总结与后续演进建议

本次重构用极小的代码代价（不到 500 行的增删），在 Streamlit 框架的极限内，实现了 **SPA (单页应用) 级别的流畅度** 和 **现代 SaaS 的版式美学**。

**未来演进方向**：
1. **持久化队列**：目前任务状态依然依赖 `.task_state.json` 文件。如果未来需要支持**真正的多任务并发**（例如同时下载 3 个视频，同时转写 2 个音频），建议将 `task_queue.py` 的底层替换为基于 SQLite 的轻量级队列库（如 `huey` 或手写表结构）。
2. **WebSocket 推送**：当 Streamlit 官方进一步完善其双向通信机制后，可将 `@st.fragment` 轮询升级为真正的服务端推送 (Server-Sent Events)。
