# 🎯 Media Tools 产品设计方案

> 从产品设计视角分析当前项目存在的问题,并提出系统性的改进方案

---

## 一、当前产品存在的问题

### 1.1 定位与价值主张问题

#### ❌ 问题 1: 产品定位模糊
**现状**: 
- 文档描述为"抖音下载+Qwen转写一体化管家",但实际是"工具集合"而非"一体化产品"
- 两个功能模块独立开发,只是简单通过 CLI 菜单拼接,缺乏真正的产品融合
- 用户心智模型混乱: 这到底是下载工具还是转写工具?

**影响**:
- 新用户难以快速理解产品核心价值
- 功能入口分散,学习成本高
- 难以形成清晰的品牌认知

**建议方案**:
```
定位重构: "自媒体内容二次创作助手"
核心价值: "一键将视频转为可编辑文字稿,加速内容创作流程"

价值主张层级:
1. 核心价值: 视频→文字稿(转写)
2. 辅助价值: 视频批量获取(下载)
3. 延伸价值: 数据分析、内容优化建议
```

#### ❌ 问题 7: 缺少工作流扩展能力
**现状**:
- Pipeline是硬编码的固定流程
- 用户无法自定义"下载后做什么"
- 无法接入外部工具(如剪映、PR、Notion)

**建议方案**:

**引入插件化工作流**:
```yaml
# workflow.yaml - 用户可自定义工作流
workflow:
  name: "视频转文稿工作流"
  trigger: "视频下载完成"
  steps:
    - action: transcribe
      config:
        format: markdown
        speakers: true
        timestamps: true
    
    - action: export
      config:
        path: ./transcripts/
        
    - action: notify
      config:
        type: wechat
        message: "转写完成!"
    
    - action: delete_original
      enabled: true

# 未来可扩展:
# - action: edit_subtitle (自动生成字幕文件)
# - action: upload_bilibili (自动发布到B站)
# - action: sync_notion (同步到Notion)
```

### 1.5 技术债务与可维护性

- 两套配置系统、两套认证系统、两套日志系统
- 公共代码重复(路径处理、错误处理、UI美化)

**建议方案**:

**中期重构计划**:
```
阶段1 - 统一基础设施(1-2周):
├── 统一配置系统 → src/media_tools/config/
├── 统一日志系统 → src/media_tools/logging/
├── 统一错误处理 → src/media_tools/errors/
└── 统一UI组件 → src/media_tools/ui/

阶段2 - 模块化重构(2-3周):
├── src/media_tools/downloader/ (抖音下载)
├── src/media_tools/transcriber/ (Qwen转写)
├── src/media_tools/pipeline/ (流程编排)

阶段3 - 标准化(1周):
├── 统一CLI入口
├── 统一测试框架
└── 统一文档规范
```

### 1.6 项目可持续性

#### ❌ 问题 9: 第三方API依赖风险
**现状**:
- 重度依赖抖音F2框架和Qwen AI API
- API一旦变更或不可用,核心功能直接瘫痪
- 缺少降级方案

**建议方案**:

**降低依赖风险**:
```
短期:
├── 关键API调用处增加异常捕获
├── 提供详细的错误诊断信息
└── 编写常见问题排查文档

中期:
├── 抽象API适配层,支持多供应商切换
├── 转写模块支持其他AI服务(如讯飞、OpenAI)
└── 下载模块支持其他平台(B站、快手)

长期(可选):
└── 本地转写模型(如whisper)作为备选
```

## 七、产品设计原则

### 7.1 核心原则

1. **用户目标优先**: 每个功能都回答"这帮用户解决了什么问题?"
2. **渐进式复杂度**: 新手3步上手,高手深度定制
3. **透明可控**: 所有自动化都可暂停、回滚、手动干预
4. **隐私安全**: 本地数据优先,云端可选
5. **可组合性**: 小工具组合,而非大而全

### 7.2 设计禁忌

- ❌ 不要替用户做不可逆的决定(如自动删除)
- ❌ 不要隐藏关键信息(如配额消耗)
- ❌ 不要让用户猜测系统状态
- ❌ 不要一次性抛出所有配置项
- ✅ 要提供"撤销"和"预览"机制
- ✅ 要用数据证明价值(节省X小时)
