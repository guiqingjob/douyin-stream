## 1. 重构背景与目标

### 1.1 现状分析

当前项目已完成核心功能开发，但随着功能迭代，代码库逐渐暴露出以下问题：

| 问题类型 | 具体表现 | 影响 |
|----------|----------|------|
| **模块职责过重** | `downloader.py` 1207行、`orchestrator.py` 871行 | 难以测试、维护困难 |
| **配置分散** | 配置来源包括 DB、文件、环境变量 | 难以追踪和管理 |
| **代码复杂度高** | 平均圈复杂度 18，部分模块超过 60 | 理解成本高、bug 率上升 |
| **测试覆盖不足** | 核心模块测试覆盖率约 60% | 重构风险高 |
| **命名不规范** | 存在 `orchestrator_v2.py` 等版本号命名 | 混淆版本概念 |

### 1.2 重构目标

| 目标 | 描述 | 预期收益 |
|------|------|----------|
| **可维护性提升** | 降低代码复杂度，提高可读性 | 维护成本降低 30% |
| **可扩展性增强** | 支持快速添加新功能 | 功能开发周期缩短 20% |
| **技术债务清理** | 消除历史遗留问题 | 技术债务评级从"中"降至"低" |
| **测试覆盖率提升** | 完善测试体系 | 覆盖率从 60% 提升到 80%+ |
| **命名标准化** | 统一命名规范 | 代码一致性提升 |

### 1.3 成功标准

- ✅ 所有现有测试通过（100%）
- ✅ 核心模块测试覆盖率 ≥ 80%
- ✅ 代码平均圈复杂度 ≤ 10
- ✅ 功能开发效率提升 20%
- ✅ Bug 修复周期缩短 30%

## 2. 重构范围

### 2.1 模块优先级矩阵

| 优先级 | 模块 | 当前问题 | 复杂度 | 影响范围 |
|--------|------|----------|--------|----------|
| **P0** | `douyin/core/downloader.py` | 1207行，职责过重 | 160 | 抖音下载全流程 |
| **P0** | `pipeline/orchestrator.py` | 871行，流程硬编码 | 64 | 转写管道全流程 |
| **P1** | `api/routers/tasks.py` | 640行，路由层包含业务逻辑 | 59 | 任务管理 API |
| **P1** | 配置系统 | 来源分散 | - | 全局 |
| **P2** | `transcribe/flow.py` | 594行，状态机复杂 | 55 | 转写流程 |
| **P2** | `api/routers/assets.py` | 437行，职责过重 | 53 | 素材管理 API |
| **P3** | `websocket_manager.py` | 缺少重连策略优化 | - | 实时推送 |
| **P3** | `logger.py` | 日志格式不一致 | - | 全局 |

### 2.2 不涉及范围

- 抖音/F2 核心 SDK 逻辑（外部依赖）
- Qwen API 调用逻辑（已稳定）
- 数据库 schema（当前设计合理）
- 前端组件（本次专注后端重构）
