# Media Tools Web 信息架构

## 目标

这个 Web 不是一个“功能堆叠后台”，而是围绕本地内容生产流程组织的工作台。

主线只有一条：

`来源 → 素材 → 文稿 → 维护`

页面必须服务这条主线，而不是各自长成一个孤岛。

---

## 一级导航结构

### 总览
- `工作台`
  - 作用：展示当前系统阶段、推荐下一步、分流到正确页面。
  - 不承担具体业务执行，只做流程路由和状态概览。

### 内容生产
- `下载中心`
  - 作用：把抖音链接或来源列表变成本地素材。
- `转写中心`
  - 作用：把素材变成文稿。
- `资产大盘`
  - 作用：按“待处理 / 已完成 / 全量”查看资产结果。

### 来源与认证
- `关注管理`
  - 作用：维护持续跟踪的内容来源。
- `账号与认证`
  - 作用：维护下载认证、转写认证、账号池与配额状态。

### 维护
- `系统配置`
  - 作用：做环境检测、配置修复、预设切换。
- `清理与备份`
  - 作用：释放空间、清理数据库记录、备份与恢复。

---

## 页面职责边界

### 1. 工作台
回答问题：**我现在该做什么？**

应该展示：
- 当前阶段
- 系统状态摘要
- 下一步 CTA
- 下载 / 转写任务摘要

不应该展示：
- 复杂配置表单
- 大量维护操作
- 全量数据表格

### 2. 关注管理
回答问题：**我在跟踪哪些来源？**

应该展示：
- 添加来源
- 来源列表
- 搜索 / 排序
- 导入导出（辅助）
- 删除来源（危险区）

不应该承担：
- 下载执行逻辑
- 认证逻辑

### 3. 账号与认证
回答问题：**系统有没有能力访问外部平台？**

应该展示：
- 抖音下载认证
- Qwen 转写认证
- 配额状态
- 账号池状态

不应该承担：
- 来源管理
- 下载/转写执行

### 4. 下载中心
回答问题：**怎么把来源变成素材？**

应该展示：
- 单链接下载
- 批量来源拉取
- 下载类当前任务
- 下载类历史任务

绝对不应该：
- 混入转写类任务状态

### 5. 转写中心
回答问题：**怎么把素材变成文稿？**

应该展示：
- 上传文件转写
- 素材库批量转写
- 转写类当前任务
- 转写类历史任务

绝对不应该：
- 混入下载类任务状态

### 6. 资产大盘
回答问题：**当前产出到了哪一步？**

应该优先展示：
1. 待处理资产
2. 已完成文稿
3. 全量资产明细

不应该一上来就只堆全表。

### 7. 系统配置
回答问题：**系统能不能稳定运行？**

应该展示：
- 环境可用性检测
- 修复与配置备份
- 预设切换

### 8. 清理与备份
回答问题：**如何安全地释放空间和维护数据？**

应该展示：
- 空间占用概览
- 数据库维护
- 备份与恢复

## 任务状态设计原则

下载与转写任务必须隔离：

- 下载任务类型：`download`, `batch_download`
- 转写任务类型：`transcribe`, `batch_transcribe`

规则：
- 下载页面只读取下载任务
- 转写页面只读取转写任务
- 首页按类型分组展示历史
- 不允许再出现“最新任务覆盖所有页面”的实现

---

## 页面结构规范

每个页面尽量遵循统一节奏：

1. 页面标题
2. 摘要指标
3. 主操作区
4. 当前状态 / 结果区
5. 辅助操作区
6. 危险操作区
7. 下一步 CTA

---

## CTA 设计原则

CTA 必须只指向“下一步最合理动作”，不要同时给太多同级按钮。

示例：
- 来源管理 → 下载中心
- 下载中心 → 转写中心
- 转写中心 → 资产大盘
- 维护页面 → 工作台 / 账号与认证

---

## 后续新增页面必须遵守的规则

新增页面前先回答三个问题：

1. 它属于哪一条主线？
   - 来源
   - 素材
   - 文稿
   - 维护

2. 它只回答哪个主问题？
   - 一个页面只回答一个核心问题

3. 它的下一步应该指向哪？
   - 页面不是终点，必须能回到工作流

如果一个页面同时承担多个主问题，就说明它应该被拆分。
