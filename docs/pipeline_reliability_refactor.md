# 媒体处理流水线可靠性重构方案

## 背景

当前项目的下载、入库、转写、账号池、重试、清理和前端任务展示等功能已经基本齐全，但生产使用中暴露出一个更高层次的问题：业务流程复杂度已经超过了当前“任务级状态 + 局部重试”的工程模型。

典型表现包括：

- 部分视频转写失败，但整体任务被标记为完成。
- 用户无法直接定位和重试未转写的视频。
- 同一个文件在失败重试过程中可能上传到多个通义账号。
- 中断或异常退出后，遗留的运行中状态可能导致后续恢复跳过视频。
- 任务状态、视频状态、数据库状态和外部平台状态之间缺少统一真相源。

因此，下一阶段的重点不应是继续堆功能，而应把系统升级为一个可恢复、可审计、可重放的媒体处理流水线。

## 核心宗旨

不要让“任务完成”代表“业务完成”。

业务是否完成应该由每个视频资产的最终状态决定；任务只代表一次执行尝试。

整个系统应围绕三个原则重构：

1. 视频级状态机
2. 任务级汇总
3. 幂等或可追踪的外部副作用

## 目标模型

### 1. 视频资产表作为业务真相源

每个视频都应该有独立、可查询、可恢复的生命周期。下载任务、转写任务、补齐任务都只是改变视频资产状态的执行过程。

建议 `media_assets` 或其关联表长期维护这些状态字段：

- `download_status`
- `transcript_status`
- `upload_status`
- `transcript_path`
- `last_error`
- `retry_count`
- `current_task_id`
- `source_platform`
- `source_id`
- `file_hash`

前端不应只依赖任务是否完成，而应该能回答这些问题：

- 某个创作者还有多少视频未转写？
- 哪些视频转写失败？失败原因是什么？
- 哪些视频可以继续处理？
- 哪些视频已经成功落盘并入库？

### 2. 子任务记录每个视频的处理结果

一个批量任务下面通常包含多个视频。任务本身不应该直接表示所有业务结果，而应由子任务聚合得出。

每个视频子任务可以有这些状态：

- `pending`
- `running`
- `success`
- `failed`
- `cancelled`

任务状态由子任务聚合：

- 全部成功：`COMPLETED`
- 全部失败：`FAILED`
- 部分失败：`PARTIAL_FAILED`
- 仍有运行中：`RUNNING`
- 用户取消：`CANCELLED`

如果短期内前端不方便支持 `PARTIAL_FAILED`，可以暂时把部分失败标记为 `FAILED`，但长期建议显式支持部分失败状态。

### 3. 转写流程拆成可恢复阶段

通义转写不是一个原子操作。它至少包含上传、开始转写、轮询、导出、下载结果、更新数据库、清理源文件等多个外部副作用。

因此，转写流程应拆为可恢复阶段：

- `queued`
- `uploading`
- `uploaded`
- `transcribing`
- `exporting`
- `saved`
- `failed`

上传成功后应持久化记录：

- `account_id`
- `record_id`
- `gen_record_id`
- `file_hash`
- `uploaded_at`
- `upload_status`

如果上传已经成功，但后续轮询或导出失败，系统应该优先使用已有 `gen_record_id` 继续轮询或导出，而不是重新上传文件。

### 4. 重试策略按错误类型和阶段决策

重试不应该简单地重新执行整个转写流程。不同错误应该有不同恢复方式：

- 认证失败：标记账号失效，切换账号。
- 网络失败：
  - 上传前失败，可以重试上传。
  - 上传后失败，只重试轮询、开始转写或导出。
- 配额失败：切换账号或等待配额恢复。
- 文件错误：直接失败，不自动重试。
- 导出失败：只重试导出，不重新上传和重新转写。
- 数据库更新失败：不删除源文件，保留可恢复状态。

这样可以显著降低重复上传、重复消耗额度和不可恢复失败的概率。

### 5. 清理必须晚于业务落盘

删除源视频只能发生在以下条件全部满足之后：

1. 转写文件已经成功下载到本地。
2. `media_assets.transcript_path` 已经更新。
3. 转写预览和全文索引已经成功或至少有明确失败记录。
4. 子任务状态已经持久化为成功。

如果任何一步失败，都不应该删除源文件。

## 分阶段落地计划

### 第一阶段：止血修复

目标是减少生产中最直接的问题。

已经完成或应优先完成：

- 任意视频转写失败时，任务不再标记为 `COMPLETED`。
- 同一视频的非认证失败不再自动跨多个通义账号重复上传。
- 历史 `running` 状态不再被恢复逻辑永久跳过。
- 前端应显示失败视频列表，并提供只重试失败视频的入口。

### 第二阶段：视频级状态治理

目标是让 `media_assets` 成为业务真相源。

建议工作：

- 下载完成后必须入库，不能只依赖 `new_files` 临时列表。
- 转写前从数据库查询待转写视频，而不是只处理当前任务返回的新文件。
- 转写成功、失败、跳过都写回视频资产状态。
- 创作者页面展示未转写、转写失败、可重试数量。
- 支持按创作者、平台、失败类型重新派发转写任务。

### 第三阶段：可恢复转写流水线

目标是让外部平台副作用可追踪、可恢复。

**已完成（2026-05-05）**：

- ✅ 新建 `transcribe_runs` 表，每行 = 某 asset 在某账号上的一次完整转写尝试，
  字段含 run_id / asset_id / account_id / task_id / stage / record_id /
  gen_record_id / batch_id / export_url / transcript_path / last_error /
  error_stage / error_type
- ✅ stage 推进序：`queued -> uploaded -> transcribing -> exporting -> downloading -> saved`
- ✅ `run_real_flow` 在 4 个关键节点写入 stage：上传完成、record/start、export
  开始、拿到 export_url
- ✅ orchestrator 每次尝试创建 run，成功 mark_saved，失败 mark_failed（错误阶段
  从 transcribe_runs 当前 stage 推断，不需手动猜）
- ✅ `find_resumable(asset_id, account_id)` 命中条件：gen_record_id 已持久化，
  且 stage 在 RESUMABLE_STAGES，**或** stage='failed' 但 error_stage 在 RESUMABLE_STAGES
  （上传成功后才挂的 run 也能被复用）
- ✅ 续传分支 1（Step 13a）：已有 export_url -> 直接 download，**0 调用 Qwen API**
- ✅ 续传分支 2（Step 13b）：已有 gen_record_id -> 跳过 token/upload/heartbeat/start，
  从 poll_until_done 继续，**节省一次 OSS 上传**
- ✅ 安全保险：续传分支任何异常 -> stage 重置 queued -> 完整 flow 接管
- ✅ asset_id 三段式 fallback：aweme 正则 -> DB 查 video_path -> 跳过 run 创建
  （孤儿文件仍能跑，只是没续传能力）
- ✅ 13 个单元 / 集成测试覆盖：建表、stage 推进、续传命中、各级 fallback、E2E
  「失败 → 重试复用 gen_record_id」

**实现的关键机制**：

- 账号池调度仍按文件状态和错误类型决策；跨账号续传**不**支持，因为 Qwen 的
  genRecordId 与账号绑定。
- 续传 fast-path 在调用 Qwen API 之前尝试，命中时连一次外部请求都不发。

**待补**（可选优化，非阻塞）：

- 失败原因聚合报表 / 健康检查（属第四阶段）
- 续传节省额度估算指标（运维可观测性）

### 第四阶段：可观测性和运维能力

目标是让问题可解释、可定位。

建议工作：

- 每个视频处理日志带上 `task_id`、`asset_id`、`account_id`、`stage`。
- 任务详情展示阶段耗时、失败阶段、最后错误。
- 增加失败原因聚合报表。
- 增加健康检查：长期 running、已下载未转写、转写成功但文件不存在、数据库状态和文件系统不一致。

## 推荐优先级

短期最重要的是：

1. 只重试失败视频。
2. 下载后确保所有视频入库。
3. 转写前按数据库状态发现未转写视频。
4. 上传成功后保存通义 `record_id` / `gen_record_id`。

中长期最重要的是：

1. 引入 `PARTIAL_FAILED`。
2. 建立视频级状态机。
3. 建立可恢复的转写阶段表。
4. 把任务系统从“业务真相源”降级为“执行记录”。

## 判断标准

重构是否成功，不看功能数量，而看系统能否稳定回答这些问题：

- 任意一个视频现在处于什么阶段？
- 为什么失败？
- 是否可以重试？
- 重试会不会重复上传、重复删除或重复入库？
- 如果服务中断，重启后能否从正确阶段继续？
- 任务结束后，是否还能准确找出业务上未完成的视频？

如果这些问题都有稳定答案，这个项目就从“功能可用”进入了“业务可靠”的阶段。
